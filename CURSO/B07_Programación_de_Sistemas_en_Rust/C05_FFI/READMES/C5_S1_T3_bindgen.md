## 1. ¿Por qué bindgen?

En el tema anterior declaramos funciones C manualmente con `extern "C"`. Esto funciona
para APIs pequeñas, pero se vuelve inviable cuando el header C tiene cientos de funciones,
structs anidados, macros, typedefs y constantes. `bindgen` automatiza este proceso: lee un
header `.h` y genera las declaraciones Rust equivalentes.

```
┌─── Manual vs bindgen ─────────────────────────────────────┐
│                                                            │
│  Manual:                                                   │
│  • Lees el header C                                        │
│  • Traduces cada función, struct, typedef a Rust           │
│  • Si el header cambia, actualizas a mano                  │
│  • Errores de firma → UB silencioso                        │
│  • Viable para ~10 funciones                               │
│                                                            │
│  bindgen:                                                  │
│  • Parsea el header C con libclang                         │
│  • Genera código Rust automáticamente                      │
│  • Si el header cambia, se regenera                        │
│  • Firmas siempre correctas                                │
│  • Viable para cualquier tamaño de API                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### ¿Cómo funciona?

```
┌─── Pipeline de bindgen ───────────────────────────────────┐
│                                                            │
│  mylib.h ──→ libclang (parser de C/C++) ──→ AST            │
│                                               │            │
│                                               ▼            │
│                                          bindgen           │
│                                          (generador)       │
│                                               │            │
│                                               ▼            │
│                                          bindings.rs       │
│                                          (código Rust)     │
│                                                            │
│  Contenido generado:                                       │
│  • extern "C" { fn ... }    para funciones                 │
│  • #[repr(C)] struct ...    para structs                   │
│  • const ...: c_int = ...   para #defines y enums          │
│  • type ... = ...           para typedefs                  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 2. Instalación y setup

### Requisitos

bindgen usa `libclang` para parsear C/C++. Necesitas instalarlo:

```bash
# Fedora/RHEL
sudo dnf install clang-devel

# Ubuntu/Debian
sudo apt install libclang-dev

# Arch
sudo pacman -S clang

# macOS (viene con Xcode)
xcode-select --install
```

### Instalar bindgen CLI

```bash
# Como herramienta de línea de comandos
cargo install bindgen-cli

# Verificar
bindgen --version
```

### Agregar bindgen como build-dependency

```toml
# Cargo.toml
[build-dependencies]
bindgen = "0.71"
```

## 3. Uso desde línea de comandos

### Generar bindings básicos

Supongamos un header C simple:

```c
// mylib.h
#ifndef MYLIB_H
#define MYLIB_H

#include <stdint.h>

typedef struct {
    double x;
    double y;
} Point;

typedef struct {
    Point origin;
    double width;
    double height;
} Rect;

int add(int a, int b);
double distance(const Point *a, const Point *b);
Rect make_rect(Point origin, double w, double h);
double rect_area(const Rect *r);

#define MAX_POINTS 1024
#define PI 3.14159265358979

#endif
```

```bash
# Generar bindings a stdout
bindgen mylib.h

# Guardar en archivo
bindgen mylib.h -o src/bindings.rs

# Con flags de compilador
bindgen mylib.h -- -I/usr/local/include -DDEBUG=1
```

### Output generado

bindgen produce código como este:

```rust
/* automatically generated by rust-bindgen 0.71.1 */

pub const MAX_POINTS: u32 = 1024;
pub const PI: f64 = 3.14159265358979;

#[repr(C)]
#[derive(Debug, Copy, Clone)]
pub struct Point {
    pub x: f64,
    pub y: f64,
}

#[repr(C)]
#[derive(Debug, Copy, Clone)]
pub struct Rect {
    pub origin: Point,
    pub width: f64,
    pub height: f64,
}

extern "C" {
    pub fn add(a: ::std::os::raw::c_int, b: ::std::os::raw::c_int)
        -> ::std::os::raw::c_int;
}

extern "C" {
    pub fn distance(a: *const Point, b: *const Point) -> f64;
}

extern "C" {
    pub fn make_rect(origin: Point, w: f64, h: f64) -> Rect;
}

extern "C" {
    pub fn rect_area(r: *const Rect) -> f64;
}
```

Observa lo que bindgen hace automáticamente:
- `#[repr(C)]` en todos los structs
- `#[derive(Debug, Copy, Clone)]` en structs que lo soportan
- `#define` convertidos a `const`
- Tipos C traducidos a `::std::os::raw::c_int` etc.
- Punteros `const *` a `*const`

## 4. Uso desde build.rs

La forma más común de usar bindgen es desde `build.rs`, regenerando los bindings cada vez
que el header cambia.

### Estructura del proyecto

```
my_project/
├── Cargo.toml
├── build.rs
├── c_src/
│   ├── mylib.h
│   └── mylib.c
├── src/
│   ├── main.rs
│   └── lib.rs     (o main.rs)
└── target/
    └── ...
        └── bindings.rs    ← generado aquí por build.rs
```

### build.rs básico

```rust
// build.rs
use std::env;
use std::path::PathBuf;

fn main() {
    // Decirle a cargo que recompile si el header cambia
    println!("cargo::rerun-if-changed=c_src/mylib.h");

    // Generar bindings
    let bindings = bindgen::Builder::default()
        // Header de entrada
        .header("c_src/mylib.h")
        // Generar los bindings
        .generate()
        .expect("Unable to generate bindings");

    // Escribir en el directorio OUT_DIR (target/debug/build/...)
    let out_path = PathBuf::from(env::var("OUT_DIR").unwrap());
    bindings
        .write_to_file(out_path.join("bindings.rs"))
        .expect("Couldn't write bindings");
}
```

### Incluir los bindings en tu código

```rust
// src/main.rs (o lib.rs)

// include! inserta el contenido del archivo generado
#![allow(non_upper_case_globals)]
#![allow(non_camel_case_types)]
#![allow(non_snake_case)]

include!(concat!(env!("OUT_DIR"), "/bindings.rs"));

fn main() {
    unsafe {
        let result = add(3, 4);
        println!("add(3, 4) = {}", result);

        let a = Point { x: 0.0, y: 0.0 };
        let b = Point { x: 3.0, y: 4.0 };
        let d = distance(&a, &b);
        println!("distance = {}", d);  // 5.0

        println!("MAX_POINTS = {}", MAX_POINTS);
        println!("PI = {}", PI);
    }
}
```

Los `#![allow(...)]` son necesarios porque los nombres generados siguen convenciones de C
(SCREAMING_CASE para constantes, snake_case inconsistente, etc.), que no coinciden con las
de Rust.

### build.rs con compilación de C incluida

Lo más común es combinar bindgen (para generar declaraciones) con cc (para compilar el
código C):

```rust
// build.rs
use std::env;
use std::path::PathBuf;

fn main() {
    // 1. Compilar el código C
    cc::Build::new()
        .file("c_src/mylib.c")
        .include("c_src")
        .compile("mylib");

    // 2. Generar bindings desde el header
    let bindings = bindgen::Builder::default()
        .header("c_src/mylib.h")
        .generate()
        .expect("Unable to generate bindings");

    let out_path = PathBuf::from(env::var("OUT_DIR").unwrap());
    bindings
        .write_to_file(out_path.join("bindings.rs"))
        .expect("Couldn't write bindings");

    // Recompilar si algo cambia
    println!("cargo::rerun-if-changed=c_src/mylib.h");
    println!("cargo::rerun-if-changed=c_src/mylib.c");
}
```

```toml
# Cargo.toml
[build-dependencies]
bindgen = "0.71"
cc = "1"
```

```
┌─── Flujo completo con build.rs ───────────────────────────┐
│                                                            │
│  cargo build                                               │
│       │                                                    │
│       ▼                                                    │
│  build.rs:                                                 │
│  ├── cc::Build compila mylib.c → libmylib.a                │
│  └── bindgen genera mylib.h → bindings.rs                  │
│       │                                                    │
│       ▼                                                    │
│  rustc compila src/main.rs                                 │
│  ├── include!(bindings.rs) inserta las declaraciones       │
│  └── link con libmylib.a resuelve los símbolos             │
│       │                                                    │
│       ▼                                                    │
│  Binario ejecutable                                        │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 5. Configuración de bindgen::Builder

### Opciones más usadas

```rust
// build.rs
fn main() {
    let bindings = bindgen::Builder::default()
        .header("c_src/mylib.h")

        // Paths de include (como -I en gcc)
        .clang_arg("-Ic_src/include")
        .clang_arg("-I/usr/local/include")

        // Definir macros (como -D en gcc)
        .clang_arg("-DNDEBUG")
        .clang_arg("-DVERSION=2")

        // Generar derives automáticos
        .derive_debug(true)          // #[derive(Debug)]
        .derive_default(true)        // #[derive(Default)]
        .derive_copy(true)           // #[derive(Copy)] cuando es posible
        .derive_eq(true)             // #[derive(Eq)] cuando es posible
        .derive_hash(true)           // #[derive(Hash)] cuando es posible
        .derive_partialeq(true)      // #[derive(PartialEq)]
        .derive_partialord(true)     // #[derive(PartialOrd)]

        // Usar tipos de core en vez de std (para no_std)
        // .use_core()

        // Layout tests: genera tests que verifican sizeof/alignof
        .layout_tests(true)  // Activado por defecto

        // Documentación
        .generate_comments(true)     // Convertir comentarios C en doc comments
        .prepend_enum_name(false)    // No prefijar variantes de enum

        .generate()
        .expect("Unable to generate bindings");

    // ...
}
```

### Tipo de enum

C enums son peculiares: son básicamente `int` con nombres. bindgen ofrece varias formas de
representarlos en Rust:

```c
// C enum
enum LogLevel {
    LOG_ERROR = 0,
    LOG_WARN = 1,
    LOG_INFO = 2,
    LOG_DEBUG = 3,
};
```

```rust
// build.rs — controlar cómo se generan los enums

// Opción 1: constantes (por defecto)
// Genera: pub const LOG_ERROR: LogLevel = 0;
//         pub type LogLevel = c_uint;
bindgen::Builder::default()
    .header("header.h")
    .default_enum_style(bindgen::EnumVariation::Consts)

// Opción 2: módulo con constantes
// Genera: pub mod LogLevel { pub const LOG_ERROR: Type = 0; ... }
bindgen::Builder::default()
    .header("header.h")
    .default_enum_style(bindgen::EnumVariation::ModuleConsts)

// Opción 3: Rust enum (verificación exhaustiva)
// Genera: #[repr(u32)] enum LogLevel { LOG_ERROR = 0, ... }
// CUIDADO: valores fuera del enum son UB
bindgen::Builder::default()
    .header("header.h")
    .default_enum_style(bindgen::EnumVariation::Rust {
        non_exhaustive: true,  // Agrega #[non_exhaustive]
    })

// Opción 4: newtype (tipo wrapper con constantes)
// Genera: #[repr(transparent)] struct LogLevel(c_uint);
//         impl LogLevel { pub const LOG_ERROR: LogLevel = LogLevel(0); }
bindgen::Builder::default()
    .header("header.h")
    .default_enum_style(bindgen::EnumVariation::NewType {
        is_bitfield: false,
        is_global: false,
    })
```

```
┌─── ¿Qué estilo de enum elegir? ───────────────────────────┐
│                                                            │
│  Consts (default):                                         │
│  + Simple, familiar                                        │
│  + Seguro: acepta cualquier valor                          │
│  - Sin type safety (todo es c_uint)                        │
│  Mejor para: la mayoría de casos                           │
│                                                            │
│  Rust enum:                                                │
│  + Type safety total, match exhaustivo                     │
│  - UB si C retorna valor no listado                        │
│  Mejor para: enums cerrados y controlados                  │
│                                                            │
│  NewType:                                                  │
│  + Type safety moderada (no confundes tipos)               │
│  + Acepta valores no listados                              │
│  - Más verboso                                             │
│  Mejor para: APIs donde mezclar enums es un riesgo         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 6. Filtrar qué se genera

Cuando el header incluye otros headers (`#include <stdio.h>`, etc.), bindgen genera
bindings para **todo**, incluyendo miles de definiciones de libc. Usa allowlist y blocklist
para controlar qué se genera.

### Allowlist: solo generar lo que necesitas

```rust
// build.rs
let bindings = bindgen::Builder::default()
    .header("c_src/mylib.h")

    // Solo generar bindings para estas funciones
    .allowlist_function("add")
    .allowlist_function("distance")
    .allowlist_function("rect_.*")     // Regex: rect_area, rect_perimeter, etc.

    // Solo estos tipos
    .allowlist_type("Point")
    .allowlist_type("Rect")

    // Solo estas variables/constantes
    .allowlist_var("MAX_POINTS")
    .allowlist_var("PI")

    .generate()
    .expect("Unable to generate bindings");
```

Con allowlist, bindgen genera solo los ítems listados **más** cualquier tipo que estos
necesiten. Si `distance` usa `Point`, se genera `Point` automáticamente.

### Blocklist: excluir lo que no necesitas

```rust
// build.rs
let bindings = bindgen::Builder::default()
    .header("c_src/mylib.h")

    // No generar bindings para estas funciones
    .blocklist_function("internal_.*")   // Funciones internas
    .blocklist_function("__.*")          // Funciones del compilador

    // No generar estos tipos
    .blocklist_type("InternalState")

    // No generar estos ítems
    .blocklist_item("DEBUG_.*")

    .generate()
    .expect("Unable to generate bindings");
```

### Ejemplo real: linkar con zlib

```rust
// build.rs — generar bindings para zlib
fn main() {
    // Solo necesitamos unas pocas funciones de zlib
    let bindings = bindgen::Builder::default()
        .header_contents("wrapper.h", "#include <zlib.h>")
        .allowlist_function("compress")
        .allowlist_function("compress2")
        .allowlist_function("uncompress")
        .allowlist_function("compressBound")
        .allowlist_type("uLong")
        .allowlist_type("Bytef")
        .generate()
        .expect("Unable to generate bindings");

    let out_path = std::path::PathBuf::from(std::env::var("OUT_DIR").unwrap());
    bindings
        .write_to_file(out_path.join("bindings.rs"))
        .expect("Couldn't write bindings");

    // Linkar con zlib
    println!("cargo::rustc-link-lib=z");
    println!("cargo::rerun-if-changed=build.rs");
}
```

### header_contents: header virtual

En vez de crear un archivo wrapper, puedes pasar el contenido del header directamente:

```rust
// build.rs
let bindings = bindgen::Builder::default()
    // Header virtual: no necesitas un archivo .h
    .header_contents("wrapper.h", r#"
        #include <math.h>
        #include <stdlib.h>
    "#)
    .allowlist_function("sqrt")
    .allowlist_function("sin")
    .allowlist_function("cos")
    .allowlist_function("abs")
    .allowlist_function("rand")
    .allowlist_function("srand")
    .generate()
    .expect("Unable to generate bindings");
```

## 7. Tipos opacos y blocklist

### Tipos opacos

Cuando C tiene un struct que solo usas como puntero (nunca accedes a sus campos), puedes
marcarlo como opaco para simplificar los bindings:

```c
// database.h
typedef struct Database Database;   // Tipo opaco en C

Database *db_open(const char *path);
int db_query(Database *db, const char *sql);
void db_close(Database *db);
```

```rust
// build.rs
let bindings = bindgen::Builder::default()
    .header("c_src/database.h")
    // Marcar Database como tipo opaco
    // bindgen genera un struct vacío en vez de intentar parsear los campos
    .opaque_type("Database")
    .generate()
    .expect("Unable to generate bindings");
```

Esto produce:

```rust
// bindings.rs generado
#[repr(C)]
pub struct Database {
    _unused: [u8; 0],
}

extern "C" {
    pub fn db_open(path: *const ::std::os::raw::c_char) -> *mut Database;
    pub fn db_query(db: *mut Database, sql: *const ::std::os::raw::c_char)
        -> ::std::os::raw::c_int;
    pub fn db_close(db: *mut Database);
}
```

### ¿Cuándo usar opaque_type?

```
┌─── ¿Opaco o completo? ───────────────────────────────────┐
│                                                            │
│  Usar opaque_type cuando:                                  │
│  • Solo usas el tipo como puntero (*mut T)                 │
│  • No necesitas acceder a campos del struct                │
│  • El struct tiene campos que bindgen no puede parsear      │
│  • El struct es privado (forward declaration en el header) │
│                                                            │
│  Generar completo cuando:                                  │
│  • Necesitas leer/escribir campos del struct               │
│  • Necesitas crear instancias del struct en Rust            │
│  • Necesitas sizeof() del struct                           │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### Blocklist de ítems complejos

A veces bindgen no puede generar ciertos tipos (templates C++, macros complejas, bitfields
problemáticos). Usa blocklist para excluirlos y declara manualmente solo lo que necesitas:

```rust
// build.rs
let bindings = bindgen::Builder::default()
    .header("complex_lib.h")

    // Este struct tiene un campo que bindgen no soporta
    .blocklist_type("ProblematicStruct")

    // Esta función usa va_list, que es complicado
    .blocklist_function("debug_printf")

    .generate()
    .expect("Unable to generate bindings");
```

```rust
// src/main.rs — declarar manualmente lo que blocklist excluó

include!(concat!(env!("OUT_DIR"), "/bindings.rs"));

// Declaración manual del tipo problemático
#[repr(C)]
pub struct ProblematicStruct {
    pub id: std::ffi::c_int,
    // Solo los campos que necesitas
}
```

## 8. Integración con cc

### Proyecto completo: header + source + bindings

```
my_crypto/
├── Cargo.toml
├── build.rs
├── c_src/
│   ├── crypto.h
│   └── crypto.c
└── src/
    └── main.rs
```

```c
// c_src/crypto.h
#ifndef CRYPTO_H
#define CRYPTO_H

#include <stdint.h>
#include <stddef.h>

typedef struct {
    uint8_t bytes[32];
} Hash256;

// Simple hash (NOT for production!)
Hash256 simple_hash(const uint8_t *data, size_t len);
int hash_equals(const Hash256 *a, const Hash256 *b);
void hash_to_hex(const Hash256 *hash, char *out, size_t out_len);

#define HASH_SIZE 32
#define HEX_SIZE  65  // 32 bytes * 2 hex chars + null

#endif
```

```c
// c_src/crypto.c
#include "crypto.h"
#include <string.h>
#include <stdio.h>

Hash256 simple_hash(const uint8_t *data, size_t len) {
    Hash256 result;
    memset(&result, 0, sizeof(result));

    for (size_t i = 0; i < len; i++) {
        result.bytes[i % HASH_SIZE] ^= data[i];
        result.bytes[(i + 1) % HASH_SIZE] =
            (result.bytes[(i + 1) % HASH_SIZE] + data[i]) & 0xFF;
    }

    return result;
}

int hash_equals(const Hash256 *a, const Hash256 *b) {
    return memcmp(a->bytes, b->bytes, HASH_SIZE) == 0;
}

void hash_to_hex(const Hash256 *hash, char *out, size_t out_len) {
    if (out_len < HEX_SIZE) return;
    for (int i = 0; i < HASH_SIZE; i++) {
        snprintf(out + i * 2, 3, "%02x", hash->bytes[i]);
    }
    out[HASH_SIZE * 2] = '\0';
}
```

```toml
# Cargo.toml
[package]
name = "my-crypto"
version = "0.1.0"
edition = "2021"

[build-dependencies]
bindgen = "0.71"
cc = "1"
```

```rust
// build.rs
use std::env;
use std::path::PathBuf;

fn main() {
    // Compilar la librería C
    cc::Build::new()
        .file("c_src/crypto.c")
        .include("c_src")
        .flag("-Wall")
        .flag("-Wextra")
        .compile("crypto");

    // Generar bindings
    let bindings = bindgen::Builder::default()
        .header("c_src/crypto.h")
        .derive_debug(true)
        .derive_default(true)
        .derive_partialeq(true)
        .derive_eq(true)
        .allowlist_function("simple_hash")
        .allowlist_function("hash_equals")
        .allowlist_function("hash_to_hex")
        .allowlist_type("Hash256")
        .allowlist_var("HASH_SIZE")
        .allowlist_var("HEX_SIZE")
        .generate()
        .expect("Unable to generate bindings");

    let out_path = PathBuf::from(env::var("OUT_DIR").unwrap());
    bindings
        .write_to_file(out_path.join("bindings.rs"))
        .expect("Couldn't write bindings");

    println!("cargo::rerun-if-changed=c_src/crypto.h");
    println!("cargo::rerun-if-changed=c_src/crypto.c");
}
```

```rust
// src/main.rs
#![allow(non_upper_case_globals)]
#![allow(non_camel_case_types)]
#![allow(non_snake_case)]

include!(concat!(env!("OUT_DIR"), "/bindings.rs"));

fn main() {
    let data = b"Hello, bindgen!";

    unsafe {
        // Calcular hash
        let hash = simple_hash(data.as_ptr(), data.len());
        println!("Hash bytes: {:?}", hash.bytes);

        // Convertir a hex
        let mut hex_buf = [0u8; HEX_SIZE as usize];
        hash_to_hex(
            &hash,
            hex_buf.as_mut_ptr() as *mut std::ffi::c_char,
            hex_buf.len(),
        );
        let hex = std::ffi::CStr::from_ptr(hex_buf.as_ptr() as *const _);
        println!("Hash hex: {}", hex.to_string_lossy());

        // Comparar hashes
        let hash2 = simple_hash(data.as_ptr(), data.len());
        let equal = hash_equals(&hash, &hash2);
        println!("Equal: {}", equal != 0);

        // Hash diferente
        let other = b"Different data";
        let hash3 = simple_hash(other.as_ptr(), other.len());
        let equal = hash_equals(&hash, &hash3);
        println!("Equal to different: {}", equal != 0);
    }
}
```

## 9. Errores comunes

### Error 1: libclang no encontrado

```
thread 'main' panicked at 'Unable to find libclang'
```

```bash
# Solución en Fedora/RHEL
sudo dnf install clang-devel

# Solución en Ubuntu/Debian
sudo apt install libclang-dev

# Si está instalado pero no se encuentra, indicar la ruta
export LLVM_CONFIG_PATH=/usr/bin/llvm-config
# O
export LIBCLANG_PATH=/usr/lib64/
```

**Por qué falla**: bindgen depende de libclang para parsear C. Sin ella, no puede analizar
el header. El paquete necesario es `clang-devel` (no solo `clang`).

### Error 2: header no encontrado por clang

```
c_src/mylib.h:3:10: fatal error: 'somelib.h' file not found
```

```rust
// MAL: no indicar los include paths
let bindings = bindgen::Builder::default()
    .header("c_src/mylib.h")  // mylib.h hace #include "somelib.h"
    .generate();               // Error: somelib.h no encontrado

// BIEN: agregar el path de includes
let bindings = bindgen::Builder::default()
    .header("c_src/mylib.h")
    .clang_arg("-Ic_src/include")          // Path local
    .clang_arg("-I/usr/local/include")     // Path del sistema
    .generate();
```

**Por qué falla**: clang necesita los mismos `-I` flags que usarías al compilar el C.
Si el header incluye otros headers, bindgen/clang necesita saber dónde encontrarlos.

### Error 3: bindings generan miles de líneas innecesarias

```rust
// MAL: sin filtros, bindgen genera TODO lo que el header incluye
let bindings = bindgen::Builder::default()
    .header("mylib.h")  // #include <stdio.h>, <stdlib.h>, etc.
    .generate();
// Resultado: bindings.rs con 10,000+ líneas de libc

// BIEN: allowlist solo lo que necesitas
let bindings = bindgen::Builder::default()
    .header("mylib.h")
    .allowlist_function("my_.*")     // Solo tus funciones
    .allowlist_type("My.*")          // Solo tus tipos
    .allowlist_var("MY_.*")          // Solo tus constantes
    .generate();
// Resultado: bindings.rs con solo tus definiciones
```

**Por qué falla**: cuando tu header hace `#include <stdlib.h>`, clang ve todas las
definiciones de stdlib, y bindgen genera bindings para todas ellas. Usa allowlist para
limitar la generación a tu API.

### Error 4: layout test failure

```
test bindgen_test_layout_MyStruct ... FAILED
assertion failed: `(left == right)`
  left: `16`,
  right: `24`
```

```rust
// Esto pasa cuando bindgen calcula un layout diferente al de C.
// Causas comunes:
// 1. Diferentes flags de compilación entre build.rs y el C
// 2. Packed structs que bindgen no detecta
// 3. Platform-specific alignment

// Solución 1: pasar las mismas flags
.clang_arg("-m64")
.clang_arg("-std=c11")

// Solución 2: si el struct usa __attribute__((packed))
// bindgen normalmente lo detecta, pero a veces falla
// Puedes desactivar layout tests para ese tipo:
.no_debug("PackedStruct")

// Solución 3: desactivar layout tests globalmente (último recurso)
.layout_tests(false)
```

**Por qué falla**: los layout tests verifican que `sizeof` y `alignof` en Rust coinciden
con los de C. Si fallan, tus bindings tienen el layout incorrecto y causarán corrupción
de memoria. No desactives los tests sin entender la causa.

### Error 5: no regenerar cuando el header cambia

```rust
// MAL: sin rerun-if-changed
fn main() {
    let bindings = bindgen::Builder::default()
        .header("c_src/mylib.h")
        .generate()
        .unwrap();
    // ...
    // Si cambias mylib.h, cargo build NO regenera los bindings
}

// BIEN: decirle a cargo cuándo regenerar
fn main() {
    println!("cargo::rerun-if-changed=c_src/mylib.h");
    println!("cargo::rerun-if-changed=c_src/mylib.c");
    println!("cargo::rerun-if-changed=build.rs");

    let bindings = bindgen::Builder::default()
        .header("c_src/mylib.h")
        .generate()
        .unwrap();
    // ...
}
```

**Por qué falla**: sin `rerun-if-changed`, cargo puede cachear la compilación y no ejecutar
`build.rs` de nuevo. El resultado es que tus bindings Rust no coinciden con el header C
actualizado → crashes, corrupción, o errores de link confusos.

## 11. Ejercicios

### Ejercicio 1: bindings a funciones de math.h

Usa bindgen para generar bindings a la librería matemática de C:

1. Crea un proyecto con `build.rs` que use `bindgen`.
2. Genera bindings para las funciones `sqrt`, `sin`, `cos`, `tan`, `pow`, `log`, `exp`,
   `ceil`, `floor` y la constante `M_PI` desde `<math.h>`.
3. Usa `allowlist_function` para filtrar solo esas funciones y `allowlist_var` para `M_PI`.
4. En `main.rs`, llama a cada función con valores de prueba y muestra resultados.
5. Agrega `println!("cargo::rustc-link-lib=m");` en build.rs para linkar con libm.

Después, inspecciona el archivo `bindings.rs` generado (en `target/debug/build/*/out/`).
¿Qué tipos usa para los parámetros y retornos?

**Pista**: usa `.header_contents("wrapper.h", "#include <math.h>")` para no crear un archivo
wrapper.

> **Pregunta de reflexión**: bindgen genera las funciones de math.h con tipo `f64` (ya que
> `double` en C es exactamente `f64`). ¿Por qué aun así es preferible usar bindgen en vez
> de declarar `extern "C" { fn sqrt(x: f64) -> f64; }` manualmente? ¿En qué caso la
> diferencia importaría?

---

### Ejercicio 2: librería C con bindgen + cc

Crea un proyecto completo que combine bindgen con cc:

1. Escribe `c_src/stats.h` con:
   ```c
   typedef struct { double mean; double stddev; double min; double max; } Stats;
   Stats compute_stats(const double *data, int count);
   void stats_print(const Stats *s);
   ```
2. Implementa las funciones en `c_src/stats.c`.
3. En `build.rs`, usa `cc` para compilar y `bindgen` para generar bindings.
4. En `main.rs`, crea un array de f64, llama a `compute_stats`, y verifica que `stats_print`
   muestra los valores correctos.
5. Agrega `cargo::rerun-if-changed` para ambos archivos C.

Verifica que si modificas `stats.c` (por ejemplo, cambiando el formato de `stats_print`),
`cargo build` recompila automáticamente.

**Pista**: usa `.allowlist_function("compute_stats|stats_print")` (regex con `|`) para
filtrar ambas funciones.

> **Pregunta de reflexión**: ¿qué pasaría si modificas el struct `Stats` en el header C
> (por ejemplo, añadiendo un campo `count`) pero olvidas ejecutar `cargo clean` antes de
> `cargo build`? ¿El `rerun-if-changed` es suficiente para detectar el cambio?

### Ejercicio 3: bindings a una librería del sistema con filtrado

Genera bindings selectivos a `<unistd.h>` (funciones POSIX):

1. Usa `bindgen` con `allowlist_function` para generar solo: `getpid`, `getppid`,
   `getuid`, `geteuid`, `gethostname`, `getcwd`, `chdir`, `sleep`.
2. Usa `allowlist_var` para `STDOUT_FILENO` y `STDERR_FILENO`.
3. Genera el `bindings.rs` y compara cuántas líneas tiene versus generar sin filtros
   (sin allowlist).
4. En `main.rs`, usa los bindings para mostrar: PID, UID, hostname, y directorio actual.

**Pista**: para `getcwd`, necesitarás crear un buffer y pasar su puntero. Revisa la firma
C: `char *getcwd(char *buf, size_t size)`.

> **Pregunta de reflexión**: sin allowlist, bindgen genera bindings para todo lo que
> `<unistd.h>` incluye (transitivamente: structs de stat, tipos de signal, etc.).
> Además del tamaño del archivo, ¿qué problemas prácticos puede causar generar
> demasiados bindings? Piensa en conflictos de nombres, tiempos de compilación, y
> mantenimiento.
